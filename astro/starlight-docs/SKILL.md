# Starlight Docs

Starlight is the official documentation theme built on top of Astro, maintained by the Astro team. This skill captures the parts of `https://starlight.astro.build` that matter when actually building a Starlight site: project setup, content authoring, the built-in component library, configuration, sidebar and search wiring, theming, i18n, and the advanced extension surfaces (overrides, route middleware, plugins).

## What Starlight is, in one breath

Starlight is an Astro **integration** (not a separate framework). You install `@astrojs/starlight`, add it to `integrations:` in `astro.config.mjs`, define a `docs` content collection that uses Starlight's `docsLoader()` and `docsSchema()`, and drop Markdown / MDX / Markdoc files into `src/content/docs/`. Every file becomes a page. The integration handles routing, layout, navigation, search (via Pagefind), syntax highlighting (via Expressive Code), and i18n.

The single most important consequence of "it's an Astro integration": when in doubt about something that isn't Starlight-specific (image optimization, deployment adapters, framework components like React or Svelte, environment variables, the dev server), the answer is in the Astro docs, not Starlight's. Don't invent Starlight-specific configuration for problems that Astro already solves.

## The minimum viable Starlight site

This is the shape every Starlight site has. If you're scaffolding something or debugging "why isn't this working," start by checking that these pieces exist and look right.

```bash
# Scaffold a fresh project
npm create astro@latest -- --template starlight
```

```js
// astro.config.mjs
import { defineConfig } from 'astro/config';
import starlight from '@astrojs/starlight';

export default defineConfig({
  integrations: [
    starlight({
      title: 'My Docs',
      // sidebar, social, logo, customCss, plugins, components, locales… all go here
    }),
  ],
});
```

```ts
// src/content.config.ts
import { defineCollection } from 'astro:content';
import { docsLoader } from '@astrojs/starlight/loaders';
import { docsSchema } from '@astrojs/starlight/schema';

export const collections = {
  docs: defineCollection({ loader: docsLoader(), schema: docsSchema() }),
};
```

```markdown
---
title: Welcome
description: A page in my Starlight site.
---

Page content goes here.
```

If a user shows you their setup and one of these three pieces is missing or wrong, that's almost always the bug — fix that before going deeper.

## Patterns worth knowing before you read further

A few cross-cutting things that come up in many situations:

- **File-based routing under `src/content/docs/`.** A file at `src/content/docs/guides/intro.md` becomes `/guides/intro/`. File names are slugified by default (lowercased, special chars stripped). The autogenerated sidebar mirrors this directory structure.
- **Frontmatter is the per-page control surface.** `title` is required; almost everything else (table of contents depth, sidebar order/label/badge, hero, draft status, search exclusion, custom `<head>` tags) is set in frontmatter. The full list lives in `references/content-authoring.md` and `references/components.md` for hero specifically.
- **Two content-syntax options beyond Markdown.** MDX (lets you import and render Astro/React/etc. components inline) and Markdoc (uses `{% tag %}` syntax, requires a separate setup step and the `@astrojs/starlight-markdoc` preset). Markdown is the default; the others are opt-in.
- **Two ways to use built-in components.** In MDX, import from `@astrojs/starlight/components` and use as JSX. In Markdoc, use the lowercased tag-style syntax (e.g. `{% card title="…" %}` instead of `<Card title="…">`). Plain `.md` files generally can't use these components — convert to `.mdx` if you need them, or use the Markdown directive equivalents (`:::note`, etc.) where available.
- **The "splash" template is for landing pages.** Set `template: splash` in frontmatter to drop the sidebars and use a wider layout, typically paired with a `hero:` block.
- **Customization escalation ladder.** When tweaking appearance/behavior, climb in this order: (1) config options in `starlight()`, (2) per-page frontmatter, (3) custom CSS via `customCss`, (4) component overrides via `components: { … }`, (5) route middleware that modifies route data, (6) a full Starlight plugin. Each step is more invasive than the last — prefer the lowest rung that solves the problem.

## A note on Starlight's release status

Starlight is in beta. Versions ship frequently and APIs occasionally shift. For features the user is depending on, point them at the [Starlight changelog on GitHub](https://github.com/withastro/starlight/blob/main/packages/starlight/CHANGELOG.md) and recommend `npx @astrojs/upgrade` (or the pnpm/yarn equivalent) when updating. If a user reports behavior that doesn't match what's in the references, a stale install is a plausible cause.
