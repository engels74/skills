# Internationalization (i18n)

Multi-language documentation: routing, fallback content, UI string translations, and full right-to-left (RTL) support.

## Basic configuration

Declare locales in the `starlight()` config. Each key is the directory name inside `src/content/docs/`; the value describes the language.

```js
// astro.config.mjs
starlight({
  title: 'My Docs',
  defaultLocale: 'en',
  locales: {
    en: {
      label: 'English',
    },
    'zh-cn': {
      label: '简体中文',
      lang: 'zh-CN',
    },
    ar: {
      label: 'العربية',
      dir: 'rtl',
    },
  },
})
```

- `label` (required) — display name in the language switcher. Write it in the language itself (`English`, `العربية`, `简体中文`) so users recognize it.
- `lang` — BCP-47 tag. Defaults to the directory name if omitted. Use this when the directory name doesn't match the tag (e.g. directory `zh-cn`, tag `zh-CN`).
- `dir` — `'ltr'` (default) or `'rtl'`.

`defaultLocale` must match one of the keys. It controls fallback content and default UI labels.

Then create a directory for each language and place content files inside:

```
src/content/docs/
├── en/
│   ├── index.md
│   └── ...
├── zh-cn/
│   ├── index.md
│   └── ...
└── ar/
    ├── index.md
    └── ...
```

For more advanced i18n scenarios (e.g. custom URL routing strategies), Starlight is also compatible with [Astro's `i18n` config](https://docs.astro.build/en/guides/internationalization/#configure-i18n-routing).

## Root locale (no language prefix)

To serve one language without a `/lang/` URL prefix, use the special key `root`:

```js
starlight({
  defaultLocale: 'root',                // optional; can omit entirely
  locales: {
    root: {
      label: 'English',
      lang: 'en',                       // 'lang' is REQUIRED for root locales
    },
    'zh-cn': {
      label: '简体中文',
      lang: 'zh-CN',
    },
  },
})
```

Files for the root locale live directly in `src/content/docs/` (not in a subdirectory):

```
src/content/docs/
├── index.md           ← English homepage at /
├── about.md           ← English page at /about/
└── zh-cn/
    ├── index.md       ← Chinese homepage at /zh-cn/
    └── about.md       ← Chinese page at /zh-cn/about/
```

### Monolingual non-English site

Starlight defaults to English. To run a monolingual site in another language, define *only* a `root` locale with the desired `lang`:

```js
starlight({
  title: 'My Docs',
  locales: {
    root: {
      label: '简体中文',
      lang: 'zh-CN',
    },
  },
})
```

This overrides Starlight's default UI language without enabling the language switcher or any other multilingual features.

## Fallback content

When you add `/en/about.md` but haven't written `/fr/about.md` yet, visitors to `/fr/about` see the English content with a notice that this page isn't translated. This makes it safe to ship the default language first and translate progressively.

To benefit from this, create files with the *same name* in each language directory. Starlight matches translations by filename, not by frontmatter.

## Translating the site title

A single `title:` string applies to every locale. For per-locale titles, pass an object:

```js
starlight({
  title: {
    en: 'My Docs',
    'zh-CN': '我的文档',
  },
  defaultLocale: 'en',
  locales: {
    en: { label: 'English' },
    'zh-cn': { label: '简体中文', lang: 'zh-CN' },
  },
})
```

## Translating Starlight's UI strings

Starlight ships with built-in UI translations (button labels, accessible labels, default aside titles, etc.) for: **Arabic, Catalan, Chinese, Chinese (Taiwan), Czech, Danish, Dutch, English, Finnish, French, Galician, German, Greek, Hebrew, Hindi, Hungarian, Indonesian, Italian, Japanese, Korean, Latvian, Norwegian Bokmål, Persian, Polish, Portuguese, Romanian, Russian, Slovak, Spanish, Swedish, Thai, Turkish, Ukrainian, Vietnamese.**

For other languages, or to override built-in strings, use the `i18n` data collection.

### Setup

1. Add `i18nLoader()` + `i18nSchema()` to the content config:

   ```ts
   // src/content.config.ts
   import { defineCollection } from 'astro:content';
   import { docsLoader, i18nLoader } from '@astrojs/starlight/loaders';
   import { docsSchema, i18nSchema } from '@astrojs/starlight/schema';

   export const collections = {
     docs: defineCollection({ loader: docsLoader(), schema: docsSchema() }),
     i18n: defineCollection({ loader: i18nLoader(), schema: i18nSchema() }),
   };
   ```

2. Create a JSON file per locale in `src/content/i18n/` (e.g. `ar.json`, `zh-CN.json`).

3. Translate the values; keep keys in English.

### Adding custom translation keys

To define your own keys (typically for use in custom components), extend the i18n schema:

```ts
// src/content.config.ts
import { defineCollection } from 'astro:content';
import { z } from 'astro/zod';
import { docsLoader, i18nLoader } from '@astrojs/starlight/loaders';
import { docsSchema, i18nSchema } from '@astrojs/starlight/schema';

export const collections = {
  docs: defineCollection({ loader: docsLoader(), schema: docsSchema() }),
  i18n: defineCollection({
    loader: i18nLoader(),
    schema: i18nSchema({
      extend: z.object({
        'custom.label': z.string().optional(),
      }),
    }),
  }),
};
```

## Using translations in components

Starlight exposes a unified translation API powered by [i18next](https://www.i18next.com/), including interpolation and pluralization.

### In Astro components

Access via `Astro.locals.t` on the global `Astro` object:

```astro
<p dir={Astro.locals.t.dir()}>
  {Astro.locals.t('404.text')}
</p>
```

### In Astro endpoints

```ts
// src/pages/404.ts
export const GET = (context) => {
  return new Response(context.locals.t('404.text'));
};
```

### In Starlight plugins

Plugins use the `useTranslations()` helper instead — see `references/plugins.md`.

### Rendering a UI string

Given a translation file:

```json
{
  "link.astro": "Astro documentation",
  "link.astro.custom": "Astro documentation for {{feature}}"
}
```

Render basic strings:

```astro
<a href="https://docs.astro.build/">
  {Astro.locals.t('link.astro')}
</a>
```

Strings with [interpolation](https://www.i18next.com/translation-function/interpolation) take an options object as the second argument:

```astro
<a href="https://docs.astro.build/en/guides/astro-db/">
  {Astro.locals.t('link.astro.custom', { feature: 'Astro DB' })}
</a>
```

### Advanced translation APIs

`t.all()` — return every UI string available for the current locale:

```astro
---
const allStrings = Astro.locals.t.all();
// { "skipLink.label": "Skip to content", "search.label": "Search", ... }
---
```

`t.exists(key, options?)` — check whether a key has a translation:

```astro
---
const keyExists = Astro.locals.t.exists('a.key');                          // boolean
const keyExistsInFrench = Astro.locals.t.exists('other.key', { lngs: ['fr'] });
---
```

`t.dir(lang?)` — get the text direction of a locale (current locale if omitted):

```astro
---
const currentDirection = Astro.locals.t.dir();      // 'ltr'
const arabicDirection = Astro.locals.t.dir('ar');   // 'rtl'
---
```

## Accessing the current locale

`Astro.currentLocale` (a built-in Astro API) gives you the active locale tag inside `.astro` files. Combined with `getRelativeLocaleUrl()` from `astro:i18n`, it's the standard way to construct cross-locale links:

```astro
---
// src/components/AboutLink.astro
import { getRelativeLocaleUrl } from 'astro:i18n';
---

<a href={getRelativeLocaleUrl(Astro.currentLocale ?? 'en', 'about')}>About</a>
```

## Things that have first-class i18n support

Most Starlight features are i18n-aware out of the box:

- **Routing** — automatic per-locale URL handling.
- **Fallback content** — see above.
- **Sidebar labels and badges** — via `translations` properties on entries (see `references/navigation-and-search.md`).
- **Site title** — see above.
- **Built-in UI strings** — many languages ship by default; rest are user-translatable.
- **DocSearch UI** — translatable via `docSearchI18nSchema`.
- **Pagefind UI** — translatable via `pagefind.*` keys.
- **RTL layout** — set `dir: 'rtl'` on the locale and Starlight handles layout flipping.
- **Per-page metadata** — translate page titles and descriptions just by writing per-locale files.
