# Content Authoring

This reference covers how content actually gets written in Starlight: the supported file formats, frontmatter fields, the Markdown extensions Starlight adds on top of the standard, and the Markdoc setup if you need it.

## Supported formats

- **Markdown (`.md`)** — works out of the box. Doesn't support importing components.
- **MDX (`.mdx`)** — works out of the box (the integration ships with the Starlight template). Supports importing and rendering components inline as JSX.
- **Markdoc (`.mdoc`)** — requires extra setup. Uses `{% tag %}` syntax for components instead of JSX. See the Markdoc section below.

Every file in `src/content/docs/` (regardless of format) becomes a page. The file's frontmatter controls metadata; the rest is content.

## Frontmatter fields

Frontmatter is YAML between `---` separators at the very top of the file. The complete list of fields:

### `title` (required)

**type:** `string`

Used as the page heading, the browser tab title, and in metadata. Always required.

### `description`

**type:** `string`

Used in `<meta name="description">` and social preview cards. Picked up by search engines.

### `slug`

**type:** `string`

Override the page's URL slug. See ["Defining custom IDs"](https://docs.astro.build/en/guides/content-collections/#defining-custom-ids) in the Astro docs.

### `editUrl`

**type:** `string | boolean`

Override the global `editLink` config for this page. Set to `false` to hide the edit link entirely, or pass a string to point it elsewhere.

### `head`

**type:** `HeadConfig[]`

Add extra tags to the `<head>` of this page only. Same shape as the global `head` option:

```yaml
head:
  - tag: title
    content: Custom about title
  - tag: meta
    attrs:
      name: og:image
      content: /custom-og.png
```

### `tableOfContents`

**type:** `false | { minHeadingLevel?: number; maxHeadingLevel?: number }`

Override the global table-of-contents config for this page. Set to `false` to disable the TOC entirely.

```yaml
tableOfContents:
  minHeadingLevel: 2
  maxHeadingLevel: 2
```

### `template`

**type:** `'doc' | 'splash'` (default: `'doc'`)

`'splash'` switches to a wider layout without sidebars. Typically used on landing pages, often paired with a `hero:` block.

### `hero`

**type:** `HeroConfig`

Render a hero section at the top of the page. Common shape:

```yaml
hero:
  title: 'My Project: Stellar Stuff Sooner'
  tagline: Take your stuff to the moon and back in the blink of an eye.
  image:
    alt: A glittering, brightly colored logo
    file: ~/assets/logo.png
  actions:
    - text: Tell me more
      link: /getting-started/
      icon: right-arrow
    - text: View on GitHub
      link: https://github.com/example/my-project
      icon: external
      variant: minimal
      attrs:
        rel: me
```

Image variants: provide a single `file:` (with optional `alt`), or `dark:` + `light:` (different image per color mode), or `html:` for raw HTML (custom `<img>` tag, inline SVG, etc.).

Action `variant` options: `'primary'` (default), `'secondary'`, `'minimal'`.

### `banner`

**type:** `{ content: string }`

Display an announcement banner at the top of the page. `content` can include HTML:

```yaml
banner:
  content: |
    We just launched something cool!
    <a href="https://example.com">Check it out</a>
```

### `lastUpdated`

**type:** `Date | boolean`

Override the global `lastUpdated` setting for this page. A YAML date overrides the Git-derived date.

### `prev` / `next`

**type:** `boolean | string | { link?: string; label?: string }`

Customize or hide the prev/next links in the page footer.

- `false` → hide the link
- `"Some text"` → keep the same target but change the label
- `{ link: '/other-page/', label: 'Custom label' }` → change both

### `pagefind`

**type:** `boolean` (default: `true`)

Set to `false` to exclude this page from the Pagefind search index.

### `draft`

**type:** `boolean` (default: `false`)

When `true`, this page is shown only in dev and excluded from production builds. Draft pages in autogenerated sidebar directories are excluded from the sidebar in production. You can't reference a draft page directly in the `sidebar` config via `slug` because it doesn't exist in production.

### `sidebar`

**type:** `SidebarConfig`

Control how this page appears in autogenerated sidebar groups:

```yaml
sidebar:
  label: Custom sidebar label  # overrides title in sidebar
  order: 2                     # lower numbers appear higher
  hidden: false                # if true, exclude from sidebar entirely
  badge:                       # add a badge next to the label
    text: New
    variant: tip
  attrs:                       # HTML attributes on the sidebar link
    target: _blank
```

`badge` can also be a plain string (uses default variant): `badge: New`.

Note: these `sidebar` frontmatter fields only affect autogenerated groups and docs items configured via `slug`. They don't apply to items configured via `link`.

## Customizing the frontmatter schema

Add your own fields (and tighten built-in ones) via `docsSchema({ extend })` in `src/content.config.ts`:

```ts
import { defineCollection } from 'astro:content';
import { z } from 'astro/zod';
import { docsLoader } from '@astrojs/starlight/loaders';
import { docsSchema } from '@astrojs/starlight/schema';

export const collections = {
  docs: defineCollection({
    loader: docsLoader(),
    schema: docsSchema({
      extend: z.object({
        // Make `description` required instead of optional.
        description: z.string(),
        // Add a new optional field.
        category: z.enum(['tutorial', 'guide', 'reference']).optional(),
      }),
    }),
  }),
};
```

To use Astro's `image()` helper (for typed image fields), pass a function:

```ts
schema: docsSchema({
  extend: ({ image }) => z.object({
    cover: image(),
  }),
}),
```

## Markdown extensions Starlight adds

Beyond standard Markdown, Starlight provides several things that aren't in vanilla Markdown.

### Asides (note / tip / caution / danger)

Wrap content in `:::` for a styled callout block:

```markdown
:::note
Starlight is a documentation toolkit built with Astro.
:::

:::tip[Did you know?]
Astro uses "Islands Architecture" for fast websites.
:::

:::tip{icon="heart"}
Custom icon (must be a name from the icon catalog).
:::

:::caution
Be careful — this could trip up your users.
:::

:::danger
Don't do this; consequences are severe.
:::
```

Custom titles go in square brackets after the type. Custom icons go in curly brackets — the value must be one of [Starlight's built-in icons](icons.md). Both can be combined: `:::tip[Did you know?]{icon="heart"}`.

The MDX/Markdoc equivalent is the `<Aside>` component, which has identical behavior but is sometimes preferred when nesting complex markup.

### Heading anchor links

Headings get clickable anchor links automatically. By default, only `<h2>` and `<h3>` appear in the page's table of contents — set `tableOfContents` in frontmatter or globally to change this.

Recommended pattern: page title comes from frontmatter `title:` (which becomes the `<h1>`). Start the page body with regular prose, then use `## ` (h2) and below for section headings. Don't put another `<h1>` in the body.

Disable anchor links globally via `markdown: { headingLinks: false }` in the integration config.

### Code blocks

Code blocks are powered by [Expressive Code](https://expressive-code.com/). You get syntax highlighting, line markers, diff-style additions/deletions, terminal/editor frames, and titles via the opening fence:

````markdown
```js {2-3}
function demo() {
  // These two lines are highlighted
  return 'with curly-brace line markers';
}
```

```js "Individual terms" /Even.*supported/
// Quoted strings and regex selectors highlight specific text.
function demo() {
  return 'Even regular expressions are supported';
}
```

```js "return true;" ins="inserted" del="deleted"
function demo() {
  console.log('These are inserted and deleted marker types');
  return true;
}
```

```diff lang="js"
function thisIsJavaScript() {
  // This block highlights as JS and supports diff markers.
-  console.log('Old code to be removed')
+  console.log('New and shiny code!')
}
```
````

Frames: shell languages (`bash`, `sh`) get a terminal-style frame; others get an editor-style frame *if they include a title*. Titles come from a `title="..."` attribute or a filename comment on the first line:

````markdown
```js
// my-test-file.js
console.log('Hello World!');
```

```bash title="Installing dependencies…"
npm install
```

```bash frame="none"
echo "This is not rendered as a terminal despite using bash."
```
````

For Markdoc syntax, options go inside `{% meta="..." %}` instead. The Expressive Code documentation has the full feature list (`https://expressive-code.com/key-features/`).

### Details / disclosures

Use plain HTML `<details>` and `<summary>` — Starlight styles them appropriately:

```markdown
<details>
<summary>Where and when is the Andromeda constellation most visible?</summary>

The Andromeda constellation is most visible in the night sky during November at latitudes between +90° and −40°.

</details>
```

### Footnotes

Standard Markdown footnote syntax:

```markdown
Here is a footnote[^1] with some additional text after it.

[^1]: My reference.
```

### Images

Two options:

- **Markdown image syntax**: `![alt text](path/to/image.png)`. Relative paths are resolved against the page file. Both relative paths (`../../assets/foo.svg`) and remote URLs work.
- **`<Image>` from Astro**: use it inside MDX files for fine-grained control. See [Astro's image guide](https://docs.astro.build/en/guides/images/).

### Inline styles

Standard Markdown applies: `**bold**`, `*italic*` or `_italic_`, `~~strikethrough~~`, `` `inline code` ``, `[link text](url)`.

## Customizing Markdown processing

`markdown: { headingLinks: false }` in the integration config disables clickable heading anchor links.

`markdown: { processedDirs: ['./src/data/reviews/'] }` extends Starlight's Markdown pipeline (asides, heading anchors, RTL code support) to additional directories — useful when rendering content from a custom collection inside `<StarlightPage>`.

To add general remark/rehype plugins, configure them on Astro's `markdown` option (top-level, not inside the `starlight()` call). See ["Markdown Plugins"](https://docs.astro.build/en/guides/markdown-content/#markdown-plugins) in the Astro docs.

## Markdoc setup

Markdoc isn't enabled by default. To add it:

```bash
npx astro add markdoc
npm install @astrojs/starlight-markdoc
```

Then create `markdoc.config.mjs` at the project root:

```js
import { defineMarkdocConfig } from '@astrojs/markdoc/config';
import starlightMarkdoc from '@astrojs/starlight-markdoc';

export default defineMarkdocConfig({
  extends: [starlightMarkdoc()],
});
```

The `starlightMarkdoc()` preset accepts:

- `headingLinks: boolean` (default `true`) — same as the global `markdown.headingLinks` option, but applies to `.mdoc` files.

After setup, `.mdoc` files in `src/content/docs/` work alongside `.md` and `.mdx`. Components use `{% tag %}` syntax: `{% card title="Stars" icon="star" %}…{% /card %}`.

You can also scaffold a fresh project with Markdoc preconfigured: `npm create astro@latest -- --template starlight/markdoc`.
